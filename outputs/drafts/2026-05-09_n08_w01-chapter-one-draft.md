# 第 1 章 — 為什麼一人公司需要 Agent 治理？

> 核心命題：**模型再強，沒有契約等於沒有交付。**

## 一、那一晚我盯著 200 頁分析報告

有一晚，一個朋友讓他剛升級的 AI Agent 跑了一輪市場研究。早上他出門時下了一句話：「幫我分析東南亞一人顧問業 2025 的市場機會。」回到家，桌面上躺著 200 頁 PDF，附帶 18 張圖、4 張競爭矩陣、12 條投資建議。

他傳給我一張截圖，問我：「這應該很值得吧？」

我打開來掃了 30 分鐘。其中**有 30 頁是幻覺**——某個叫做「VietWise Consulting」的公司不存在，引用的「2024 越南顧問業協會白皮書」沒有那個協會。**有 20 頁偏離了他今早的目標**——他要的是「東南亞」，AI 自己決定把印度也算進去。**還有 5 條投資建議直接和他三個月前下過的決策衝突**——AI 不知道那段歷史。

那一晚他做了什麼？他熬夜把 200 頁刪到 60 頁，再花 4 小時去查那些公司是否真的存在。他算了一下：**比他親手做還慘**——親手做的話，至少他知道哪些是事實、哪些是推論。

問題出在哪？模型不弱。Claude / GPT / Gemini 在 2026 已經比 2024 的人類分析師更會寫長報告。

問題是**他沒有跟 Agent 簽合約**。沒有交付規格、沒有可驗收的完成標準、沒有「你今天到底花了我多少錢」的紀錄。他把一個沒有 PM、沒有 QA、沒有合約的「外包」放到家裡跑了 8 小時。

他當然會收到一份很厲害但用不了的東西。

## 二、一人公司沒有的三件事

如果你是一家有規模的公司，這套東西其實已經內建好了。

你有 PM，會幫你把模糊的「分析東南亞市場」拆成「列出三個目標市場、每個市場 5 家標竿、引用必須來自 2024 年後的公開資料」。你有 QA，會在交付前抽 10% 抽查事實。你有 Reviewer，會問一句「你引用的那個白皮書連結貼一下」就把幻覺攔下來。

但**一人公司沒有這三件事**。

不是因為你不想要——是因為你雇不起。市場研究員一人月薪 8 萬、QA 工程師一人月薪 7 萬、Reviewer 兼職一個月也要 5 萬。你決定一個人開公司的那天，這三個職位就被你**砍給 LLM** 了。

問題是 LLM 預設不會做這三件事。它預設會討好你——你說「分析」，它就分析；你說「再多一點」，它就再生 50 頁。它不會問你「你的 ICP 是誰」、不會在第 17 個小時告訴你「我已經 web search 失敗 3 次，剩下 60 頁是我憑訓練資料補的」、不會在你下週問同樣問題時，說「上次我們已經做過這個分析，要不要直接用那份？」

它不知道，**因為你沒有告訴它怎麼記得**。

這就是「治理」的入口。治理不是給大組織的奢侈品——大組織反而最不需要治理，他們有 layer-by-layer 的人力把治理做進日常會議。**治理是給一人公司的保險**。你雇不起 PM/QA/Reviewer，那你就要把這三件事**用一份契約寫進 Agent 的工作流**。

不寫，每一次失誤都會變成你下個月的客訴。寫了，每一次失誤都至少能溯源。差別不在 AI 多聰明，差別在你能不能說清楚「我這個月到底花了 27 小時跑了哪些任務、有 3 件失敗了、失敗的原因可以分到哪一類、我要不要補救」。

## 三、本書要給你的三件事：可恢復、可審計、可量化

這本書要教你寫的契約，落在三條原則上。

### 可恢復

**「你開錯方向時，能不能倒車？」**

LLM 跑長任務時，最怕的不是失敗，是**靜默的偏移**。它不會告訴你「我從第 3 段開始已經在編了」，它只會繼續寫到第 200 頁。

可恢復的核心是：**任何一個關鍵階段都要能 git commit checkpoint**。失敗時你能回退到上一個正確點，而不是把 8 小時全部丟掉重跑。

這對應一條失敗模式：**COORD-01「context 被重置」**——session 中斷、tool 呼叫失敗、模型自己把上下文壓縮掉。沒有 checkpoint，每一次重啟你都從零開始。

### 可審計

**「三個月後，你看得懂為什麼當初這樣做嗎？」**

這是一人公司最常忽略的痛點。你今天做了一個決策——「這個客戶不接、那條產品線砍掉」——三個月後客戶問你「為什麼當初拒絕」，你可能已經想不起來。

可審計的核心是：**Audit Log + Decision Log**。每一張 Task Card 結束都有結構化紀錄（task_id、目標、用了哪些工具、花了多少 token、是否需要核准、最終結果），每一個跨 session 的決策都有單獨的 yaml 檔（決策日期、選項、判準、結果、復盤觸發條件）。

這對應兩條失敗模式：**SPEC-01「目標違反」**——你下個月不記得當初目標是什麼，重做還是會再偏一次；**SPEC-04「不知何時停」**——你不記得「這件事我們上週就決定不做了」。

讓我用一個真實例子說明這條原則為什麼**不是清高**。

我自己的 harness 跑過一張卡叫 `20260404-S01`，目標是「分析 AI 時代一人公司最具長遠獲利潛力的商業項目」。任務交付了：DoD 六條全部通過、產出在 `outputs/drafts/ai-era-solo-business-strategy.md`。

如果你只看這份產出，會覺得這次任務很乾淨。

但 Audit Log 多了一行 `error_summary`：「第 3 次 web search 遭遇速率限制（rate limit），以前兩次搜尋結果及既有知識完成任務。」

——**沒有那一行，這件事就不存在**。我下個月翻回來，會以為這份分析的事實基礎是 3 次 web search，實際上是 2 次 + LLM 訓練資料補完。下次我用同樣方法跑類似任務、撞同樣的 rate limit、又用同樣方式補完，**我不會察覺自己已經在踩同一個雷三次了**。

Audit Log 不是給 PM 看的。是給三個月後忘記這一切的你看的。

### 可量化

**「你今天花的每一塊錢，對得到一個產出嗎？」**

LLM 是世界上最容易花掉錢卻看不見的東西。一個普通的「幫我分析一下」可能在背後跑 80 次 tool 呼叫、消化 25K token、燒掉 2 美金。十次這樣的「幫我分析一下」，一個月就是 600 美金。一年 7,200 美金。

可量化的核心是：**每張 Task Card 結束都有 estimated_tokens / actual_tool_calls / approval_needed**。不只是事後查帳，而是事前預算——一張卡開始前你就決定「這件事最多 5 次工具呼叫、最多 3K token，超出就停」。

這對應一條失敗模式：**SEC-03「成本失控」**——這條失敗很少被列在「LLM 失敗模式」清單裡，因為它不是技術錯誤。它是**沒有契約**的副作用。你給了 Agent 一張無上限信用卡，它當然會刷爆。

## 四、為什麼一人公司比大組織更需要這套

大組織失誤了，會有 layer 吃下來——PM 漏單，QA 補；QA 漏抽，Reviewer 補；Reviewer 漏一次，下季 retro 還能修正。

**一人公司失誤了，下個月就是客訴**。

這不是嚇你。這是基本算術。一個人服務 5 個客戶，5 個客戶各代表你 20% 的營收。一個失誤、一個客訴，就是你 20% 的當月現金流。一年只能撞 3-4 次。

你能讓 AI 自由發揮、賭它每次都不出錯嗎？也可以。但那叫**信任**。

這本書的標題告訴你下一條路：**用契約取代信任**。

不是因為 AI 不可信，而是因為**你雇不起一個專門幫你管 AI 的人**。所以你只能用契約——一份你今早寫下來、Agent 跑完之後可以逐條驗收的合約——把那個不存在的 PM/QA/Reviewer 換成一份 5KB 的 yaml。

下一章開始，我會把這三條原則拆成 9 個可以今天就做的動作。每一個動作都是我自己的一人公司在過去兩個月裡跑出來、留下 18 筆 audit、6 筆 decision log 的真實流程——不是理論，是已經發生過的工作紀律。
