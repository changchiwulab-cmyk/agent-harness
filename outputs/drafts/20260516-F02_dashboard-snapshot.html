Decision Timeline

2026-04-03｜20260403-D001
Task Card YAML + git checkpoint
一人公司規模下，可恢復性（任何時間點都能 git checkout 回到任意狀態） 比即時查詢能力更重要。YAML 純文字可直接被 agent 讀取注入 context， 無需額外 API 呼叫。git 已在工具鏈中，維護成本為零。
2026-04-03｜20260403-D002
粗略護欄 + 事後量測
v1 目標是「跑起來並累積實測數據」。精確 token 追蹤在沒有基準數據前意義不大； 先累積 audit log，再用實測平均值校正預算（即 RETRO 流程所做的事）。 複雜度留給 v2+ 再引入。
2026-04-15｜20260415-D003
維持 v2，下次 retro（累積 5 筆後）再評估
四項觸發條件均未達標：context 未超限、無規則衝突、 token 雖超預估但未超 dashboard 上限、錯誤率僅 1/8（非阻斷性）。 一人公司原則：不做沒有實際需求的架構升級。
2026-04-15｜20260415-D004
升為 allow（PERMISSIONS.yaml 中移入 allow 區塊）
歷史數據明確：8/8 筆任務均為 low risk 且立即 approve，ask 無實質把關。 GATE_POLICY 的 risk_check 是第二道防線，確保即使 Task Card 建立時無確認， 執行後仍會驗證 risk_level 與實際動作一致。 選擇直接 allow（而非條件升）是因為：Task Card 本身是分析輸出，不是危險操作； 真正的風險在執行階段，而非建立 Task Card 階段。
2026-04-15｜20260415-D005
加入快速路徑判斷（fast-path），保留完整流程作為備援
事後補填（reasoning 明註原因）：本決策在 2026-04-15 實作 retro 建議時已執行， 但未同步建立 Decision Log。2026-04-17 補建以保留決策可追溯性。 依據： 1. 8/8 實測資料（100% 快速路徑），顯示完整流程當前不必要 2. 但無法排除未來「委外任務」「非本人需求」等模糊情境 3. 保守原則：既有機制不刪除，僅補充判斷邏輯 4. 與 D004（create_task_card 升 allow）思路一致 —— 依實測降低摩擦，但保留上層安全網（GATE_POLICY）
2026-04-24｜20260424-D006
選 C（Narrow Scope）
「可恢復、可審核、可量化」三原則中，量化最需要詳細資料； 但近 5 筆任務零失敗零 partial，顯示 happy-path 下 AUDIT_LOG 已充分。 failed/partial 任務才是需要深度 post-mortem 的場景。 保留 schema 但限縮使用範圍，既不失能力又不強加儀式負擔。
2026-05-09｜20260509-D007
- Repo 命名：agent-governance - License：Apache-2.0 - Visibility：Private（v0.1.0 內部驗證後再轉 public） - Layout：獨立 repo - Bootstrap 走法：本 session 在 outputs/drafts/agent-governance-bootstrap/ 下完整準備檔案樹（plugin.json + 5 commands + 4 schemas + 2 hooks + 2 validators + LICENSE + README + CHANGELOG + CI workflow）；下個 session 由使用者執行：(1) 在 GitHub 新建 private repo agent-governance；(2) git init 並把 outputs/drafts/agent-governance-bootstrap/* 全數搬入；(3) push v0.1.0 tag。
四個子題答案均選與 plan §6 + N4 §1.3 解耦聲明對齊的選項。 Apache-2.0 偏好為日後與企業整合保留 patent 條款（MIT 也可，但 Apache 多 patent 保護成本接近於零）。 Private 起步是為了讓 v0.1.0 → v0.2.0 之間的 API breaking change 不被外部使用者卡住； v0.2.0 穩定後手動轉 public。
