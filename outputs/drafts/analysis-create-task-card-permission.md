# 評估：create_task_card 是否應從 ask 升為 allow

**Task Card**：20260415-A01  
**日期**：2026-04-15  
**Skill**：analysis

---

## 結論與建議

**建議選項 B：條件升為 allow（明確需求 + 低風險任務）**，而非全面升或維持現狀。

理由：8 筆歷史任務全部立即 approve，表明現有 ask 已無實質把關作用；
但保留高風險任務的確認機制仍有價值。條件升級可兼得效率與安全。

---

## 高風險假設

- **「低風險 = 無需確認」**：若使用者未來在 `risk_level: low` 的任務中塞入實際上中/高風險的行動，條件升級的保護失效。
  → 緩解：GATE_POLICY 的 risk_check 是第二道防線，即使 Task Card 建立時沒有確認，執行後仍會驗證 risk_level 與實際動作一致性。

- **「歷史 8 筆全 approve = 未來也會 approve」**：若任務性質改變（例如開始做外發任務），現在設的預設值可能不合適。
  → 緩解：選項 B 的條件判斷確保高風險任務仍走確認流程。
