## 1. 主軸命題（已決定使用 T1；T2/T3 為對照組）

> **決定：採用 T1。** T2/T3 在 v2 改寫，作為對照角度保留——確認 T1 在這兩個對立面下仍最強。

### 候選 T1（採用）
> **「用契約取代信任：一人公司讓 LLM Agent 不失控的三條治理原則。」**

切入點：把 LLM 視為要管理的系統，不是要崇拜的工具。三原則 = 可恢復／可審計／可量化。

### 候選 T2（v2 — 外包視角，未採用）
> **「把 LLM 當外包做：一人公司用合約、簽收單、紀錄本管 Agent。」**

切入點：把 Agent 視為供應商。Task Card = 合約，DoD = 簽收單，Audit/Decision Log = 紀錄本。對「請過爛外包」有過痛感的讀者共鳴強。
不選的原因：類比過度具體會綁住讀者群——沒外包經驗的讀者進不來。但作為書中第 3 章的章內譬喻很好用。

### 候選 T3（v2 — 時間軸視角，未採用）
> **「治理比模型更不過時：給三年後的自己留下讀得懂的紀錄。」**

切入點：與「等模型變強就好了」的論述正面對抗——但用「未來的你會看不懂今天的決定」這個直觀痛點切入，而不是抽象的「治理層」概念。
不選的原因：論述偏防禦（賣「不會壞」），不如 T1 主動（賣「主動掌控」）。但作為第 12 章「移植性」的收尾命題很合適。

### 為什麼最終仍選 T1（對照後的判斷）

- **抽象層級剛好**：T2 太具體（外包），T3 太抽象（時間軸）；T1 的「契約 / 信任」是一個熟悉但有反差的概念對。
- **承載章節最廣**：T2 框死在「外包關係」，T3 框死在「未來自己」；T1 可同時承載「為什麼治理 / 如何治理 / 三原則」三條主線。
- **行銷張力對等**：「用契約取代信任」一句口號可獨立流通；T2 需要「LLM 是外包」這個前提先成立、T3 需要讀者已有「未來困擾」的自覺。
- **與差異化定位（§5）契合**：LangChain 文件偏工具、Chip Huyen 偏工程、Anthropic 官方偏 best practice，「治理／契約思維」這個切口空缺最大。
- **實證撐腰**：三原則（可恢復／可審計／可量化）有 Harness 兩個月實證（D001–D006、18+ 筆 audit），不是口號而是已落地的紀律。

### T2/T3 的二次用途（不浪費）

- T2 的「合約 / 簽收單 / 紀錄本」三詞組 → 第 3 章 Task Card 開場譬喻
- T3 的「三年後的自己」框架 → 第 12 章「治理思想可移植到哪裡」收尾段

## 2. 章節大綱（12 章）

> 每章含：標題、核心命題、內容摘要、預估字數（中文）、Harness 實證案例。

### 第 1 章 — 為什麼一人公司需要 Agent 治理？
- 核心命題：模型再強，沒有契約等於沒有交付。
- 摘要：開場用一個反例——「很厲害」的 LLM 一晚上可以幫你產 200 頁分析，但如果其中 30 頁是幻覺、20 頁偏離了你今早的目標，你比親手做還慘。一人公司沒有 QA、沒有 PM、沒有 Reviewer，所有失誤都會變成你下個月的客訴。治理不是給大組織的奢侈品，是給一人公司的保險。第二段提出本書三原則。
- 字數：3,500
- 實證：`20260404-S01` web search rate limit 後仍宣稱 done（DoD 6/6 通過，但 error_summary 留下了「第 3 次速率限制」紀錄）—— 沒有 audit log 這件事就消失了。

### 第 2 章 — 三原則總覽：可恢復 / 可審計 / 可量化
- 核心命題：三件事，每件對應一條失敗模式的解法。
- 摘要：可恢復解 COORD-01「context 被重置」、可審計解 SPEC-01/SPEC-04「目標違反/不知何時停」、可量化解 SEC-03「成本失控」。先講三原則的對位，再帶出本書其餘章節的對應關係表。
- 字數：3,000
- 實證：`system/FAILURE_TAXONOMY.yaml` 14 種失敗、四大分類（規格 42% / 協調 37% / 驗證 21% / 安全獨立維度）。

### 第 3 章 — Task Card：契約優於指令
- 核心命題：每個任務有 `goal` + `definition_of_done`，不然不執行。
- 摘要：對比兩種起手式——「幫我寫一份提案」vs「Task Card：goal=…, DoD 7 條」。第一種是許願，第二種是合約。本章拆解 Task Card 的 5 個必填欄位（goal / DoD / risk_level / allowed_tools / expected_output），每個欄位回應一條失敗模式。重點：DoD 一定要可逐條 pass/fail，不能是「品質好」這種模糊話。
- 字數：4,500
- 實證：`tasks/2026-04-04_ai-era-solo-business-proposal.yaml`（DoD 7 條全部通過）→ `2026-04-04_ai-era-solo-business-proposal-review.yaml`（找到 3 條必須修改、6 條建議修改）→ `2026-04-04_proposal-fix-v2.yaml`（修正後 DoD 5/5 通過）。三張卡的接力是一人公司 end-to-end review 流程的活樣本。

### 第 4 章 — Definition of Done：把模糊變成可驗收
- 核心命題：DoD 是一人公司唯一的驗收方。寫得糊，就驗收糊。
- 摘要：DoD 寫法的反面教材（含糊／太多／重複目標）vs 正面寫法（可機器檢核／逐條獨立／包含「不做什麼」）。Gate Policy 的 `completion_check` 要求逐條 pass/fail，這是把 DoD 從口號變成檢查項的關鍵。並提出寫 DoD 的「三條檢驗」：每一條能否獨立否決交付？能否 1 分鐘內判斷？是否包含格式與內容兩種維度？
- 字數：3,500
- 實證：`tasks/2026-05-09_harness-v3-governance-extraction.yaml` DoD 7 條全部以動詞開頭、每條都可逐項 pass/fail（如「列出 v2 全部 16 模組，每個明確標註：保留／砍除／抽出／重構」）。對比 archived 中 WEEKLY_REVIEW_TEMPLATE 早期模糊條款。

### 第 5 章 — Audit Log：你今天做了什麼？三個月後你說得出來嗎
- 核心命題：紀錄不是給 PM 看的，是給三個月後的自己看的。
- 摘要：Audit log 的目的不是合規，是讓未來的自己能 reconstruct 當下的判斷。本章給出 audit log 的最小欄位（task_id / date / goal / status / tools_called / output_path / notes / estimated_tokens），並示範如何用 audit log 做事後的 cost-quality 比較分析。
- 字數：3,000
- 實證：`logs/AUDIT_LOG.md` 中 `20260502-T01/T02/T03` 三筆是同一主題不同投入（1/3/5 web searches，~9K/24K/42K tokens）的對照組，事後可量化「多投入是否買到對應價值」。沒有結構化 audit，這條經驗就會消失。

### 第 6 章 — Decision Log：把「為什麼這樣決定」存起來
- 核心命題：決策不是動詞，是一份檔案。
- 摘要：每個 ≥medium 風險的選擇都該寫一張 Decision Log（context / options / decision / reasoning / risk / revisit_trigger）。一人公司最容易犯的錯是「我之前決定了什麼來著？」——重複糾結、左右搖擺、對不起昨天的自己。本章示範一個完整的 Decision Log 樣本，並提出「決策可追溯」與「決策可逆轉」的差別。
- 字數：3,500
- 實證：`memory/active_projects/agent-harness/decisions/20260424-D006_execution-log-scope.yaml` —— 對 Execution Log Schema 落地率低的問題，提了 A/B/C 三方案、選 C（Narrow Scope）、列了 risk 與 revisit_trigger。是「決策不是一拍腦袋而是一份檔案」最完整的範例。

### 第 7 章 — Failure Taxonomy：失敗不是意外，是有名字的
- 核心命題：把失敗分類了，才能不一直撞同一面牆。
- 摘要：14 種 LLM Agent 常見失敗（規格 42% / 協調 37% / 驗證 21% / 安全獨立維度），每種有 ID、描述、緩解措施。當 audit log 中 error_summary 出現時，要能引用 ID（例如「SEC-03 成本失控」「COORD-02 模糊需求硬做」）。失敗 ID 是治理層的詞彙表。
- 字數：3,000
- 實證：`system/FAILURE_TAXONOMY.yaml` 14 條全部來自 v1→v2 過程中累積的真實事故，不是預設清單。`20260404-S01` 的 error_summary（rate limit）對應 SEC-03。

### 第 8 章 — 「現階段不做清單」：一人公司最強的紀律工具
- 核心命題：寫下不做什麼，比寫做什麼更省時間。
- 摘要：README 的「現階段不做清單」（多 agent swarm / 自動長期記憶 / 太多工具串接 / 自動外發 / 把所有規則塞進單一 system prompt 等 8 條）是 Harness 兩個月維持精瘦的關鍵。本章方法論：每個版本都要有「不做清單」，並要附「升級觸發條件」——什麼條件達成才能拿出來重議。
- 字數：3,000
- 實證：`README.md` v1→v1.5→v2→v3→v4 的版本表，每階段都有觸發條件；D003「V3 upgrade hold」的決策本身就是「不做清單」紀律的體現——明明可以做，因為條件未達就不做。

### 第 9 章 — Gate Policy：四層驗證的成本/效益
- 核心命題：不是每個任務都要四層；驗證也有成本。
- 摘要：Schema → 規則 → 完成 → 風險，四層各有失敗處理（重試 / 立即停 / 列缺漏 / 回 drafts/）。本章要破除「驗證越多越好」的迷思——D006 的決定就是把 Execution Log Schema 從「全任務必寫」收斂到「fail/partial/high-risk/多 checkpoint 才寫」，因為對 happy-path 任務而言，audit log 已充分。**驗證的力度應與任務風險匹配**。
- 字數：3,500
- 實證：D006 Decision Log（Narrow Scope 抉擇）+ `system/GATE_POLICY.yaml` 的 rollback 欄位（每層失敗都有可回退的具體動作）。

### 第 10 章 — 強制力結構：prompt 約束、CI 護欄、runtime hook
- 核心命題：規則寫在哪裡，決定了它有多強。
- 摘要：三個層級的 enforcement——文字規則（最弱、易違反）、CI 護欄（中等、commit 時擋）、runtime hook（最強、執行前擋）。Harness v2 的進化路徑就是把核心規則一條一條從 prompt 抬到 CI、再從 CI 抬到 hook。一人公司對應的策略：高頻違反的規則一定要抬到 hook，低頻的留 prompt 就好。
- 字數：3,500
- 實證：`scripts/permissions_guard.py`（PreToolUse hook，把 PERMISSIONS deny 改 runtime 攔截）+ `scripts/check_context_budget.rb`（CI 限制 CLAUDE.md+GLOBAL_RULES ≤3K token）+ `.github/workflows/spec-consistency.yml`。對應 Phase A enforcement 卡 `20260502-A01`。

### 第 11 章 — Retrospective：從 20+ 筆 audit 找出 3 個你不知道的自己
- 核心命題：定期 retro 不是儀式，是把累積的 audit 變成可下決策的訊號。
- 摘要：方法論——每 5–10 筆任務做一次小 retro，找：(1) 重複出現的失敗模式 ID、(2) 與 Decision Log 不一致的實際做法、(3) 該晉升為 reports/ 的草稿。retro 的產出是 Decision Log，不是備忘錄。
- 字數：3,000
- 實證：`outputs/reports/retro-2026-Q2-01.md`（首次 retro 的正式報告）+ `tasks/2026-04-17_evidence-gap-filling.yaml`（retro 發現 3 項證據空白後反向開的補洞任務）。

### 第 12 章 — 收尾：治理思想可移植到哪裡
- 核心命題：契約三件是 LLM 用例之外也能跑的東西。
- 摘要：DoD / Audit / Decision Log 三件，本來就是早於 LLM 的工程治理思想（合約測試、變更紀錄、ADR）。Harness 只是把它們搬到 LLM 用例。也就是說，這套方法論不會因為模型升級而過時。最後給讀者一張「明天就能用」的 minimal kit：5 欄位 Task Card 模板 + 7 欄位 audit 紀錄 + 5 欄位 Decision Log。
- 字數：3,000
- 實證：D001「Task Card over Database」（決策原因即「結構化決策不需要重型工具」）+ Harness 自身就是 12 個月內可拆換 runtime（從 v1 單代理 → v3 治理 plugin）但思想不變的活樣本。

**總字數估計**：~40,000 字（書版本可擴充至 60,000；部落格系列版可拆 12–16 篇；課程版每章 30–45 分鐘）。

## 4. 三種發布形態取捨

| 形態 | 優勢 | 風險 | 投入 | 變現節奏 |
|------|------|------|------|---------|
| 書（紙本+電子） | 權威、長尾、可被引用 | 寫作週期 6–12 月；題材若脫節難改版 | 高（~600 小時） | 慢（出版後 1–2 年） |
| 部落格系列 | 快速迭代、可由讀者反饋導向、SEO 紅利 | 易被視為散文、難形成資產感 | 中（~150 小時，分 12 篇） | 中（每篇即時，3 個月見複利） |
| 線上課程 | 互動最強、單價高、可帶社群 | 製作門檻最高（影音/腳本/平台） | 高（~400 小時，含錄製） | 快（上線即收入） |

### 推薦順序

1. **部落格系列（先做）**：12 篇對應 12 章，每週 1 篇。低成本快驗證主軸命題、累積讀者反應、為書與課程蓄水。
2. **線上課程（次做）**：用部落格已驗證的主軸 + 真實提問做課程綱要；3–4 章合一個模組，6 模組共 8–12 小時。
3. **書（最後）**：等部落格累積 6–8 篇 + 課程跑過 1 期，再以三者最強內容定稿。書是「沉澱」不是「起點」。

理由：一人公司資源有限，書是高沉沒成本；部落格是低沉沒成本可學習的市場訊號收集器。**先學市場再寫書**。

## 5. 差異化定位

| 對標物 | 它的定位 | 差異 |
|--------|---------|------|
| **LangChain 文件**（含 LangSmith / LangGraph 文件）| 工具導向：怎麼用我家的 chain / agent / observability | 我們不教用工具，教**怎麼把任何工具納入治理框架**。LangChain 是給工程師讀工具文件，本書是給一人公司讀治理思想 |
| **《AI Engineering》(Chip Huyen, 2024)** | 系統工程導向：模型評估、資料管線、部署、監控 `[待驗證 — 以公開摘要為依據，未逐章核對]` | 對象不同：Chip 寫給 ML 平台 / 工程組；本書寫給一人公司／極小團隊。範圍不同：Chip 涵蓋訓練到部署全鏈；本書聚焦使用端的治理與驗收 |
| **Anthropic 官方 best practice**（含 prompt engineering、agent design） | 廠商視角：怎麼從 Claude 拿到最好結果 | 我們站在使用方：假設模型會錯、會漂移、會升級，**外圍治理層才是穩定的**。Anthropic 教你寫 prompt，本書教你**沒有理想 prompt 也能交付** |

獨特切入點（一句話）：
> **「市面上的書教你怎麼讓 LLM 做更多，這本書教你怎麼讓 LLM 不做你沒簽收的事。」**

## 6. 預備兩個讀者反對意見

### 反對 1：「治理太重，一人公司不需要這麼多儀式」
**回應**：
- 重點不在儀式，在**極小可運作組合**：Task Card 5 欄位、Audit 7 欄位、Decision Log 5 欄位 = 17 個欄位，5 分鐘填完。
- 反問：你今天做的事，三個月後說得出來嗎？說不出來，省下的儀式時間都會在「我之前到底決定了什麼」上吐回去。
- 第 12 章的 minimal kit 就是給「我先試 30 天」的讀者用的版本。
- 實證：D006 收斂 Execution Log Schema 為「窄範圍」恰恰是治理思想自我糾正的範例——治理越重越好，恰恰**反**治理思想。

### 反對 2：「LLM 進步後這些都會被自動處理」
**回應**：
- 三原則來自更早的工程實踐（合約測試 80 年代、變更紀錄 70 年代、ADR 2010 年代），跨過了多次模型/語言/平台的世代更替仍存活。
- 模型升級會讓**做事**變便宜，不會讓**簽收**消失。一人公司不是因為 LLM 笨才需要簽收，是因為**任何外包都需要簽收**。
- 反例論證：2026 年模型確實大幅進步，但 Harness 自身的 audit/decision/DoD 結構不僅沒被淘汰，反而從 v1 → v2 → v3 都被保留並加厚（A01 規劃「砍 5、抽出 6」，抽出的全都是治理三件 + Failure Taxonomy）。
- 邊界承認：模型升級**會**淘汰我們的「執行平面」（skill/router/permission 重疊度 70–85%，A01 表 A），但治理層（governance plane）不會。
