# 模型調度守則 DELEGATION_PLAYBOOK

> 讀者：主對話（指揮官），不論你是哪個模型。
> 目的：把「什麼事自己做、什麼事派誰做、怎麼驗收」變成查表就能執行的規則。
> 建立：2026-07-03（Fable 5 session）。事實性內容已對照官方文件查證，來源標註於各節。
> 修改本檔前先讀 `system/MAINTENANCE_PROTOCOL.md`。

## 0. 一頁摘要

- 指揮官不下場：大量讀取、掃 repo、web search、批次改檔一律派 subagent，主對話只進結論。
- 派工必附三件套：目標與動機、驗收條件、回報格式。
- 顯式指定 model 與 effort，不要用預設值碰運氣。
- 回報合約：subagent 只回結論 + 檔案:行號；長產物落檔傳路徑。
- 驗證不自驗：驗收派 fresh-context agent。
- 升降級有路徑：小模型錯一次升級；中模型同一子任務連錯兩次帶失敗軌跡升級；解出的模式降回便宜模型套用；同一件事最多重試兩輪。

## 1. 模型現實表（2026-07-03 查證）

| 別名 | 對應模型 | 在本 harness 的角色 |
|------|---------|-------------------|
| `opus` | Opus 4.8（`claude-opus-4-8`） | **指揮官**（主對話預設）；困難判斷的升級終點 |
| `sonnet` | Sonnet 5（`claude-sonnet-5`） | **預設工人**：實作、研究、審查、重構 |
| `haiku` | Haiku 4.5（`claude-haiku-4-5-20251001`） | **批次工人**：大量讀取、格式轉換、機械套用 |
| `fable` | Fable 5 | 本環境長期不使用（僅 2026-07-03 制度建設 session 用過一次）；除非使用者明說，不要指定 |

指定方式（來源：code.claude.com/docs/en/sub-agents.md、model-config.md）：

- **單次派工**：Task/Agent tool 呼叫時傳 `model` 參數（接受別名或完整 ID）。這是本 harness 的標準做法——每次派工顯式指定。
- **effort 參數**：subagent 定義檔（`.claude/agents/*.md`）frontmatter 支援 `effort: low | medium | high | xhigh | max`。若當前環境的派工工具不支援單次呼叫傳 effort，就在 prompt 內明確描述深度要求（例如「只要定位不要分析」＝low 的效果）。
- **優先序**：環境變數 `CLAUDE_CODE_SUBAGENT_MODEL` > 單次呼叫參數 > frontmatter `model` > 繼承主對話模型。注意：若使用者設了該環境變數，你的 model 參數會被蓋掉——派工後發現模型不對，先檢查這個。
- **內建 agent 類型**：`Explore`（唯讀搜索）、`Plan`（規劃）、`general-purpose`（全工具）。預設繼承主對話模型，可用 model 參數覆寫。
- 主對話模型用 `/model` 切換；訂閱方案不同預設不同（Max 預設 Opus）。

### effort 預設對照

| 工作性質 | model | effort（或 prompt 等效描述） |
|---------|-------|------------------------------|
| 定位檔案、關鍵字掃描 | haiku | low——「只回位置，不要分析」 |
| 批次格式轉換、機械套用已定義的模式 | haiku | low–medium |
| 讀多檔並總結、一般研究 | sonnet | medium |
| 實作、重構、除錯 | sonnet | high |
| 驗收審查 | sonnet | medium——附明確 checklist 時 medium 就夠 |
| 架構取捨、模糊題、二次失敗後的重攻 | opus | high–xhigh |

## 2. 指揮官不下場

主對話的 context 是最貴的資源：它一滿就壓縮，壓縮就丟目標。所以粗活外包，主對話只留結論。

### 量化門檻（碰到任一條就派工，不要自己做）

| 動作 | 門檻 | 派給 |
|------|------|------|
| 讀檔案 | 預計要讀 **> 5 個檔**，或單檔 **> 500 行**且只需要其中一部分 | Explore（唯讀） |
| 掃 repo / 找「某東西在哪」 | 不確定位置、需要試多個關鍵字 | Explore |
| web search | **任何** web search（原文絕不進主對話） | general-purpose |
| 批次改檔 | 同一模式要改 **> 3 個檔** | general-purpose（haiku，模式先在主對話定義好） |
| 產長文（報告、章節草稿） | 預計產出 **> 200 行** | general-purpose（sonnet），落檔回傳路徑 |
| 驗收 | 一律 | fresh-context agent（見 §6） |

### 例外（自己做，不派工）

- 已知確切路徑、只讀 1–3 個小檔。
- 單點修改（1–2 個檔、位置明確）。
- 與使用者的往返釐清、以及要動用 ask/deny 權限的判斷——**權限判斷永遠不外包**。
- 派工本身的成本會超過任務本身（例如改一行字）。

判斷猶豫時：估「這件事的中間產物會不會超過 100 行進 context」——會，就派工。

## 3. 派工三件套（缺一不派）

每個派工 prompt 必含三塊。填空模板見 `system/DELEGATION_TEMPLATES.md`。

1. **目標與動機**：做什麼＋為什麼（動機讓 subagent 在邊界情況能做對取捨）。
2. **驗收條件**：可勾稽的完成判準，直接抄或改寫自 Task Card 的 definition_of_done。「找出所有用到 X 的地方」不合格；「列出每個呼叫 X 的檔案:行號，含測試檔，找不到就明說找不到」合格。
3. **回報格式**：明確規定回什麼、多長、什麼格式（見 §4 回報合約）。

沒寫驗收條件的派工 = 你在賭 subagent 猜得中你要什麼。它猜不中。

## 4. 回報合約

- subagent 只回：**結論 + 證據位置（檔案:行號）+ 明確的失敗聲明**（哪些沒找到/沒做到）。
- 長產物（> 50 行）一律寫到檔案，回傳路徑。研究筆記進 `outputs/drafts/`，暫存物進 scratchpad。
- **禁止**把原始搜尋結果、整檔內容、網頁原文貼回主對話。
- 指揮官收報告後：只把「會影響後續決策的結論」留在對話裡，其餘用路徑引用。
- subagent 的回報不可信任為已驗收——它說做完了 ≠ 做完了（見 §6）。

## 5. 升降級路徑

| 情況 | 動作 |
|------|------|
| haiku 工人輸出錯誤或不合格 **1 次** | 不要重試 haiku——同樣的料它會犯同樣的錯。直接把同一任務＋錯在哪升給 sonnet |
| sonnet 工人**同一子任務**連錯 **2 次** | 帶完整失敗軌跡（兩次的 prompt、輸出、錯誤點）升給 opus 或主對話親自做。不帶軌跡的升級會重蹈覆轍 |
| opus / 主對話也解不動 | 停。這超出調度能解的範圍——依 `system/JUDGMENT_RUBRICS.md` §何時停下來問使用者 |
| 難題解出來了、剩下是同模式批次套用 | **降級**：把解法寫成明確模式（改什麼、怎麼改、一個做完的範例），派 haiku/sonnet 批次執行 |
| 同一件事累計重試 | **最多兩輪**。第三次不是換模型的問題，是方法錯了——回 JUDGMENT_RUBRICS §方向錯了的訊號 |

計數規則：「同一子任務」以驗收條件為準——驗收條件沒變就是同一子任務，換個說法重派也算重試。

## 6. 驗證不自驗

產出者的 context 帶著「我剛做完它」的偏誤，看不見自己的錯。驗收一律派 **fresh-context agent**（新開的 subagent，不給它執行過程，只給驗收材料）。

| 產出類型 | 驗收方法 |
|---------|---------|
| 檔案/文件 | **read-back**：驗收 agent 重讀檔案，逐條比對 definition_of_done，每條標 pass/fail＋證據（檔案:行號） |
| 程式碼 | **測試或實跑**：跑既有測試、或實際執行改動路徑並貼輸出。「看起來對」不算過；沒有可跑的驗證方式時，明說是靜態審查 |
| 高風險判斷（risk ≥ high、要給使用者做決策的結論） | **第二意見**：另派一個 agent 不看第一份結論獨立做一次，比對分歧；或**多答案評審**：產 2–3 個候選，派評審 agent 依明確標準選優並說明淘汰理由 |

驗收 agent 的 prompt 只給：驗收條件清單、產出路徑、必要的背景一句話。**不給**執行過程、不給「我覺得已經完成」之類的暗示。

驗收 fail 的處理回到 `system/GATE_POLICY.yaml` 的 on_fail 規則。

## 7. 環境差異對照

本 harness 會在兩種環境運行，共通規則如上；差異只有這些：

| | 本機 Claude Code CLI | 遠端（claude.ai/code 等） |
|--|---------------------|--------------------------|
| GitHub 操作 | `gh` CLI | `mcp__github__*` 工具（無 gh）；push 後開 **draft PR** |
| 分支 | 依使用者慣例 | 只能推 session 指定的 `claude/...` 分支 |
| 自訂 agent | `.claude/agents/*.md`（frontmatter 可設 model/effort/tools） | 同樣讀 repo 內 `.claude/agents/`；另有平台內建 agent 類型 |
| 網路 | 依本機設定 | 走 agent proxy，遇 TLS/403 看 `/root/.ccr/README.md`，不要關閉驗證 |
| 暫存檔 | 專案外目錄 | 用 session 提供的 scratchpad 路徑，不用 /tmp |

拿不準當前是哪個環境：看有沒有 `gh` 指令可用（`which gh`）。
