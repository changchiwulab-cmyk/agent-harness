# JUDGMENT_RUBRICS — 判斷力外化判準

> 用途：把原本靠「模型夠強才會做對」的判斷，變成照著走就能做對的 checklist。
> 讀者：任何等級的 session（含 haiku）。每組 rubric 附一個正例、一個反例。
> 這裡管「怎麼判斷」；判斷完要派工／升級的操作步驟在 `DISPATCH_POLICY.md`。

---

## §1 何時該升級模型

**判準（任一命中 → 升級，按 DISPATCH_POLICY §5 的階梯）：**

- [ ] 同一子任務已失敗達門檻（haiku 1 次；sonnet 同題 2 次）
- [ ] 產出連續兩版「都對但都不好」——驗收 pass 卻被指出平庸/不到位
- [ ] 任務需要跨 >3 個檔案的因果推理（改 A 會不會壞 B 和 C）
- [ ] 題目是取捨而非執行：兩個方案各有代價，需要權衡而不是查表
- [ ] 自己發現在「猜」：同一句話改了 3 遍還不確定哪版對

**不升級的情況：** 失敗原因是 prompt 缺脈絡（先補三件套重派同級）；失敗原因是
外部環境（rate limit、檔案不存在）；純體力活的量很大（量大≠難，照舊用便宜模型分批）。

**正例：** sonnet 兩次重構都讓測試綠但 verifier 指出「介面耦合沒真的解開」。這是
取捨題（怎麼切介面），帶完整失敗軌跡升 opus。✔
**反例：** haiku 批次改檔失敗一次，原因是 prompt 沒給檔案清單。升級到 sonnet 重派。✘
——這是 prompt 缺件，補上清單重派 haiku 即可；升級解決不了缺脈絡。

## §2 何時算真的完成

**判準（全部滿足才准報完成）：**

- [ ] Task Card 的 definition_of_done **逐條**有 pass/fail 標記——由 fresh-context
      verifier 標的，不是自己標的（DISPATCH_POLICY §6）
- [ ] 每個 pass 都有證據：檔案:行號、測試輸出、或實跑結果——「我檢查過了」不是證據
- [ ] 產出檔案 read-back 過：真的存在、非空、內容是最終版而非中間版
- [ ] 該跑的檢查跑了且綠（本 repo：`check_spec_consistency.rb`、`check_context_budget.rb`、
      相關 `test_*`；改了 tasks/logs/decisions 的 YAML → `generate_frontend_manifest.py`）
- [ ] 對照 goal 問一次：「使用者拿到這個，他的原始問題解了嗎？」——DoD 有時寫窄了，
      DoD 全過但 goal 沒解 → 回報差距，不硬宣告完成

**正例：** 報告寫完 → verifier 逐條比對 DoD 引行號 → CI 綠 → 回報「7/7 pass，
證據如下」。✔
**反例：** 「我已完成報告撰寫並存檔」——沒有 verifier、沒有逐條、沒有 read-back。
這正是 FAILURE_TAXONOMY VAL-01（假完成）。✘

## §3 何時該停下來問使用者

**判準（任一命中 → 停下來問，用選項題不用開放題）：**

- [ ] 兩個合理解讀會導向**不同的產出**（例：「整理客戶名單」= 去重？還是重新分類？）
- [ ] 要動的東西在 PERMISSIONS `ask`/`deny` 級，或 risk_level ≥ high
- [ ] 做到一半發現實際工作量/範圍是 Task Card 估計的 2 倍以上
- [ ] 需要品味裁決：語氣、命名、對外文案的風格——rubric 驗不了的（見 §5 極限段）
- [ ] 發現 Task Card 的 DoD 本身有錯或互相矛盾

**不問的情況（自己決定，把決定寫進回報）：** 可逆的實作細節（變數名、段落順序）；
有既定慣例可查的（checkpoint 格式、目錄結構——查檔案，不要問）；
任一合理選擇成本差異可忽略的。

**問法規範：** 給 2–4 個具體選項＋各自後果＋你的建議與理由。被問的人 10 秒內能答。

**正例：** 「名單有 37 筆重複但電話不同。A：保留最新（建議，last_contact 較可信）
B：合併保留兩支電話 C：列出待你逐筆定。」✔
**反例：** 「請問名單想怎麼整理呢？」——把思考丟回給使用者。✘
**反例 2：** 變數該叫 `client_list` 還是 `customer_list` 也去問。——可逆細節，自己定。✘

## §4 方向錯了的訊號（換路，不是重試）

重試修「執行失誤」；換路修「方法錯誤」。連續失敗時先分類再行動：

**「方向錯了」的訊號（任一命中 → 停止重試，換方法或上報）：**

- [ ] 兩次失敗的**錯法不同**（第一次 A 壞、修好後 B 壞）——在打地鼠，代表對系統的
      理解模型錯了，繼續打只會更碎
- [ ] 每次「修好一點」但離 DoD 的距離沒變近（改動量遞增、進度不動）
- [ ] 開始需要越來越多特例/補丁才能讓方案成立
- [ ] 發現在對抗工具或框架（跟 linter/schema/CI 鬥智），而不是在解使用者的問題
- [ ] 方案依賴一個沒驗證過的假設，而該假設驗證一次就能推翻（先去驗假設！）

**「只是執行失誤」的訊號（重試同路可以）：** 錯誤訊息明確指出具體錯處；改一處
即綠；失敗來自環境抖動（網路、rate limit）。

**換路的動作：** 退回最後一個綠的 checkpoint（`RECOVERY_RUNBOOK.md` 場景 B）→
寫下失敗方法與原因（防下個 session 重蹈）→ 換一個**方法**（不是換措辭重試）→
無第二條路可想時依 §3 上報使用者。

**正例：** 修 schema 錯 → 修完換 manifest 漂移錯 → 修完換測試錯。錯法一直變，
stop：read 檢查器源碼弄懂全部約束，一次改對。✔
**反例：** 同一個 YAML 欄位錯，重試 5 次每次換個寫法碰運氣。第 2 次就該打開
validator 源碼看它到底要什麼。✘

## §5 品質底線怎麼驗

品質分兩層。**可驗層**用下面的 checklist；**不可驗層**（品味）誠實外包。

**可驗層（每項產出交 verifier 前先自查，verifier 再獨立查一次）：**

- [ ] 事實主張有來源；推論標明依據；不確定的標 `[待驗證]`（research skill 格式）
- [ ] 檔內引用的路徑/指令/欄位名，實際存在且拼寫正確（逐一驗過，不是眼熟）
- [ ] 數字經得起重算（總數、百分比、日期差）
- [ ] 對外文件零佔位符（沒有 TODO、`[填入]`、lorem ipsum 殘留）
- [ ] 格式匹配既有慣例（跟同目錄的既有檔比對過結構）
- [ ] 反向測試：故意找一個「這產出答不了的相關問題」——答不了是否 DoD 範圍外？
      範圍內答不了 = fail

**不可驗層（品味、策略取捨、語氣）——harness 的誠實極限：**
rubric 全過只保證「不差」，不保證「好」。弱模型無法自知平庸。凡產出屬於：
對外定稿文案、策略建議、高風險決策文件，rubric 過了之後**仍必須**走三選一：
(1) `opus` 第二意見（DISPATCH_POLICY §6.2）(2) 使用者過目（本檔 §3）
(3) 產出上標注「品質僅驗證至 rubric 層，品味未驗」。禁止略過標注直接交付。

**正例：** 產業研究報告：來源逐條可點、數字重算過、verifier 反向測試「主要競品的
定價模式」發現報告沒答而 DoD 有 → fail 打回。✔
**反例：** 「報告寫得很完整、結構清晰、應該沒問題」——三個形容詞，零證據。✘
