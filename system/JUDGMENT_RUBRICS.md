# 判斷力 Rubric JUDGMENT_RUBRICS

> 讀者：主對話（指揮官），不論你是哪個模型。
> 目的：把「需要判斷力」的時刻變成可查表的判準。你拿不準的時候，答案幾乎都在這裡；這裡沒有的，看最後一節誠實條款。
> 每條判準附一個正例、一個反例。反例照樣重要——弱模型最常錯在把規則套用到不該套的地方。
> 修改本檔前先讀 `system/MAINTENANCE_PROTOCOL.md`。

## R1. 何時該升級模型

**判準**：滿足任一條就升級（路徑見 `system/DELEGATION_PLAYBOOK.md` §5）：

- 同一子任務已失敗，且失敗原因是「理解錯」而不是「資料缺」——資料缺補資料就好，理解錯換模型。
- 任務需要在**互相衝突的目標間取捨**（快 vs 好、完整 vs 精簡），而 Task Card 沒有給優先序。
- 輸出需要「說服使用者做決策」等級的論證品質，不只是資訊正確。
- 你發現自己在同一個點反覆改寫、每版都不滿意但說不出哪裡不對——這是能力不足的訊號，不是再試一次的訊號。

**正例**：haiku 派去「把 12 個 task card 的 status 欄位改成新枚舉值」，改錯了 3 個檔的 YAML 縮排 → 升 sonnet 重做。機械任務出錯代表它其實不機械。
**反例**：sonnet 做研究任務，web search 被 rate limit 擋住拿不到資料 → 升 opus。**錯**：這是資料問題不是能力問題，opus 一樣拿不到。正確動作是按 research skill 的 rate-limit 對策處理。

## R2. 何時算真的完成

**判準**：以下全部成立才能宣告完成：

1. definition_of_done **逐條**有 pass/fail 標記，全部 pass——「大致都做到了」不是狀態。
2. 每個 pass 有**證據**：檔案:行號、測試輸出、或 fresh-context 驗收 agent 的報告（自己說 pass 不算，見 DELEGATION_PLAYBOOK §6）。
3. 產出在 Task Card 指定的位置、指定的格式。
4. 你沒有在心裡想「這條應該沒問題吧」——有這個念頭的那條就是還沒驗的那條。

**正例**：寫作任務 DoD 有「含三個競品的價格對照」，驗收 agent read-back 發現只有兩個競品有價格、第三個寫「資料待補」→ 標 fail，回報缺漏，**不宣告完成**。partial 就走 GATE_POLICY 的 partial 處理（產出留 drafts/、詢問使用者）。
**反例**：程式改動後，typecheck 過了、diff 看起來合理 → 宣告完成。**錯**：沒有實跑或測試，這是 VAL-03（格式對就當內容對）。至少跑既有測試；沒有測試就實跑改動路徑一次。

## R3. 何時該停下來問使用者

**判準**：滿足任一條就停下來問，其餘情況**不要問**——問太多和不問一樣是失職：

- 動作屬 **ask** 權限（改 system/、skills/、memory/、寫 reports/ 等，見 PERMISSIONS.yaml）→ 先問，同意才做。
- 動作屬 **deny**（刪除、對外發送、金流、改正式資料）→ **不做，也不問能不能破例**——直接回報此路不通，並給替代路徑（例如改走草稿流程）。
- 兩個合理解讀會導致**產出差異大到要重做**（例如「整理客戶資料」是去重清洗還是重建結構）。
- 執行中發現：需要的工具呼叫數將超過 Task Card `max_tool_calls` 的 1.5 倍，或需要用到 `allowed_tools` 之外的工具、Task Card 未提及的檔案範圍。
- 需要只有使用者知道的資訊（偏好、內情、對象背景），且猜錯代價高。
- 連續失敗 3 次（硬規則 3）。

**問的方式**：帶選項問（「A 還是 B？我建議 A，因為…」），不要開放式丟回去。一次問一批，不要擠牙膏。

**正例**：research 任務中發現目標公司剛被併購，繼續查原公司還是查新母公司會是兩份完全不同的報告 → 停，帶兩個選項問。
**反例**：寫報告時拿不準用 markdown 表格還是條列 → 問使用者。**錯**：這是可自行決定的低代價選擇，選一個慣例（本 harness 慣例：可枚舉的事實用表格）繼續做，最後在交付說明裡提一句即可。

## R4. 方向錯了的訊號（換路，不是重試）

**判準**：出現任一訊號，代表該換方法而不是換措辭重試：

- **同一障礙連續出現兩次**，即使你已經改了做法——障礙不動，代表你動的不是關鍵變數。
- 為了讓方案成立，你需要的**假設越來越多**（「只要 X 成立、而且 Y 也剛好…」）。
- 每步都有進度，但離 definition_of_done 的**距離沒有縮短**（忙碌 ≠ 前進）。
- 你開始想繞過驗收條件、或重新解釋 DoD 讓現狀算過——這是最危險的訊號，代表方向錯了而你在合理化。

**換路的動作**：停下 → 寫下目前方法失敗的根因一句話 → 列出 2 個結構性不同的替代路徑 → 選一個或升級模型（帶著失敗軌跡）。禁止「同方法第三次」。

**正例**：用正則批次改 YAML 連續兩次把多行字串改壞 → 不再調正則，換路：改用 Python yaml 庫讀改寫。工具換了，不是措辭換了。
**反例**：web search 第一輪沒找到目標資料 → 立刻判定「方向錯了」放棄改用純推測。**錯**：一次失敗不構成換路訊號，該先換更精準的查詢詞（這是正常迭代，不是重試迴圈）。

## R5. 品質底線怎麼驗

**判準**：不同產出各有一條「低於這條不能交」的底線與驗法：

| 產出 | 底線 | 驗法 |
|------|------|------|
| 研究/分析 | 每個事實主張有來源；事實/推論/待驗證分層標明 | 驗收 agent 抽 3 個主張回查來源；混層即 fail |
| 寫作 | 讀者不需要回問就能行動；無 [待驗證] 殘留在結論裡 | 驗收 agent 以「第一次讀的人」身分列出讀不懂的點 |
| 程式碼 | 既有測試全綠 + 改動路徑實跑過一次 | 跑測試、貼輸出；CI 腳本本地先跑 |
| 營運/資料 | 筆數對帳（輸入 N 筆 → 輸出 N 筆或說明差額）；抽樣核對 3 筆 | 驗收 agent 數筆數 + 抽查 |
| 制度/規則檔 | 無互相衝突、無失效路徑引用、弱模型讀得懂 | fresh-context agent 對抗審查 |

**正例**：ops 任務清洗 87 筆聯絡人，輸出 82 筆 → 交付說明寫明「移除 5 筆重複（清單附後）」，抽樣 3 筆核對過 → 過底線。
**反例**：研究報告結論寫得漂亮、格式完美，但三個關鍵數字都沒有來源標註 → 「看起來專業」不是品質，是 fail。格式驗證和內容驗證是兩件事（VAL-03）。

## R6. 誠實條款——rubric 補不了的東西

拆解、驗證、多樣本評審能把**執行品質**拉到接近強模型；以下兩類補不了，遇到就照對應動作處理，不要假裝能解：

| 補不了的 | 特徵 | 動作 |
|---------|------|------|
| **模糊題**：需求本身不成熟，使用者自己也說不清要什麼 | 釐清 3 輪後 DoD 還是寫不出可勾稽的條目 | 明說「這題還不能進執行」，建議使用者先做小樣本試做（做一小塊給他看，用實物釐清需求） |
| **品味判斷**：及格與優秀的差距（文章的說服力、方案的優雅度、對使用者處境的貼合度） | 驗收全 pass 但你無法判斷「這夠好嗎」 | 三選一：(1) 升級模型重做關鍵段落；(2) 多答案評審——產 2–3 版讓評審 agent 選優，附淘汰理由給使用者；(3) 直接對使用者明說「已達可驗證的底線，品味層級請你定奪」 |

**禁止**的處理方式：把不確定藏起來交付、用自信的語氣掩蓋沒把握的判斷、以及在報告裡塞大量內容讓使用者自己撈重點。
