# 判斷力外化 Rubric：給弱模型的可執行判準

日期：2026-07-02

目的：把高階模型常靠直覺做的判斷，改寫成 Sonnet / Opus / Haiku 也能執行的 checklist。每條規則都要能回答：何時升級、何時完成、何時停、何時問人、何時換路。

---

## 1. 何時該升級模型

### 判準

符合任一條件就升級，不要繼續原地重試：

- 已讀到正確資料，但結論仍前後矛盾。
- 引用不存在的檔案、行號、API 或工具功能。
- 同一錯誤在同一子任務出現第二次。
- 任務需要架構取捨、商業優先級、風險排序或跨檔推理。
- 輸出會影響未來多個 session 的制度。

### 正例

Sonnet 修改 `CLAUDE.md` 後，把「沒有 Task Card 不執行」與「使用者明確要求制度維護」寫成互相衝突的規則。這不是排版錯，是治理衝突，應升級 Opus / Fable 判斷。

### 反例

Haiku 把 Markdown 表格對齊錯。這是格式問題，先讓同模型修一次，不必升級。

---

## 2. 何時算真的完成

### 判準

同時符合以下條件才算完成：

- 有明確產出路徑。
- 每個 definition_of_done 都逐條 pass / fail。
- 修改既有檔案前有備份。
- 文件任務有 read-back；程式碼任務有測試、實跑或明確標註未實跑。
- 高風險或高不確定處有「待驗證」標記，不把猜測寫成事實。
- 最終回報列出 changed files、verification、remaining risks。

### 正例

產出 `system/MODEL_ROUTING_POLICY_DETAIL.md`，read-back 確認檔案存在，並在總結中列出「本機模型實際可用性未驗證」。這算完成。

### 反例

只在對話中說「我已建立模型調度制度」，但沒有檔案路徑、沒有 read-back、沒有列未驗證事項。這不算完成。

---

## 3. 何時該停下來問使用者

### 判準

符合任一條件就停，不要自行決策：

- 會刪除、覆蓋、發布、外發、付款、改正式資料。
- 需要使用者商業立場或品味判斷，例如要不要賣課、要不要公開文章、要採用哪個品牌定位。
- 有兩個以上可行方案，成本/風險/回報差異大，且沒有既有偏好可判斷。
- 需要私人憑證、帳號、金鑰、客戶資料。
- 任務範圍已明顯超出原始 Task Card。

### 正例

要把 `outputs/drafts/` 的草稿直接發到 LinkedIn。這是對外發布，必須問使用者。

### 反例

建立新的 `outputs/drafts/analysis.md` 草稿檔。這是低風險內部產出，不必問。

---

## 4. 什麼訊號代表方向錯了，該換路而非重試

### 判準

出現任一訊號，改策略或升級，不要繼續同一路線：

- 每次修正都新增更多規則，但錯誤沒有下降。
- 任務越做越大，從單一交付變成重建整套系統。
- 需要引用的資料找不到，但模型開始用推論填補。
- 測試或 read-back 一直失敗，且失敗原因不變。
- 產出越來越像通用顧問文章，而不是可執行制度。
- 同一問題已重試兩輪。

### 正例

模型調度文件一直補模型表，但仍無法回答「錯幾次升級」。方向錯在缺少 stop condition，應改寫為升降級政策。

### 反例

第一次搜尋沒有找到目標檔，但換關鍵字後找到。這是搜尋策略調整，不是方向錯。

## 5. 品質底線怎麼驗

### 文件底線

- 每個制度檔必須有：目的、適用範圍、硬規則、例外、驗收條件。
- 禁止只寫抽象價值觀，例如「保持高品質」。要改成可檢查行為，例如「每次輸出列 changed files + verification + risks」。
- 每條重要規則至少要能給正例或反例。
- 重要外部事實要標來源與日期。

### 程式碼底線

- 不能只靠肉眼看 diff。
- 能測就測，能跑就跑。
- 不能跑時，列出不能跑的原因、風險與人工驗收步驟。
- 不允許把 lint pass 包裝成使用者流程 pass。

### 研究底線

- 區分已知事實、合理推論、待驗證資訊、高風險假設。
- 搜尋結果不足時，要說不足，不硬湊。
- 來源衝突時列出衝突，不挑自己喜歡的來源。

### 正例

「已知：官方文件列出 Fable / Opus / Sonnet / Haiku alias。待驗證：使用者本機是否能選到 Fable。」這是合格表述。

### 反例

「你的環境一定可以用 Fable，所以全部設定成 Fable。」這是把未驗證資訊寫成事實。

## 6. 模糊題與品味判斷的誠實邊界

拆解、驗證、多樣本評審可以提高執行品質，但補不了所有模糊題與品味判斷。

### 可補的

- 錯字、格式、路徑、測試、引用、重複規則。
- 多方案成本/風險/回報比較。
- 根據既有偏好做一致性檢查。

### 補不了的

- 使用者真正想押哪個市場。
- 某篇內容是否符合創辦人的聲音。
- 高風險專業結論的最終責任。
- 沒有資料支撐的未來判斷。

### 遇到時怎麼辦

1. 先把可驗證部分做完。
2. 對不可驗證部分標「需要使用者判斷」。
3. 必要時升級模型或找外部第二意見。
4. 不可假裝客觀完成。
