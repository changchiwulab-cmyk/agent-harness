# 派工 Prompt 模板 DELEGATION_TEMPLATES

> 讀者：主對話（指揮官）。派 subagent 時直接複製對應模板，填掉全部 `【】` 空格。
> 每份模板已內建三件套（目標與動機／驗收條件／回報格式）與回報合約。規則出處：`system/DELEGATION_PLAYBOOK.md`。
> 填空原則：**寧可多給一句背景，不要讓 subagent 猜**。subagent 看不到你的對話歷史，你沒寫的它一概不知道。

---

## T1. 搜尋／定位（agent: Explore ｜ model: haiku ｜ effort: low）

適用：找「某東西在哪」、盤點某模式的所有出現位置。不適用：需要理解程式邏輯的深度追查（那是 T4 研究）。

```
目標：在 【repo/目錄範圍】 找出 【要找什麼，具體到字串/模式/概念】。
動機：【找到之後要拿來做什麼——這決定它要不要回報邊界案例】。

範圍：只搜 【目錄/檔案類型】；排除 【如 _archive/、node_modules 等】。

驗收條件：
- 列出每一個命中位置：檔案路徑:行號 + 該行內容一行
- 含測試檔與設定檔（除非上面排除）
- 一個都沒找到就明說「零命中」，並列出你試過的 3 種搜法

回報格式：條列，每行「路徑:行號 — 一句話說明」。總長 ≤ 30 行；
超過 30 個命中就寫到 【落檔路徑】，只回摘要 + 檔案路徑。
禁止貼回大段原始檔內容。
```

## T2. 實作（agent: general-purpose ｜ model: sonnet ｜ effort: high）

適用：按明確規格寫新功能/新檔案。前提：你已經想清楚要什麼——規格模糊就先回 JUDGMENT_RUBRICS R3。

```
目標：實作 【功能一句話】。
動機：【為什麼要做、誰會用、用在哪個流程】。

規格：
- 輸入：【】 輸出：【】
- 檔案位置：【新增/修改哪些檔，明確路徑】
- 遵循既有慣例：參考 【一個現成的類似檔案路徑】 的風格與結構
- 邊界情況：【空輸入/錯誤輸入怎麼處理】

驗收條件（做完自己先跑一遍，但最終驗收另有 fresh agent）：
- 【可執行的驗證：跑哪個測試/指令、預期輸出什麼】
- 既有測試全綠：【測試指令】
- 不改動規格之外的檔案

回報格式：改動檔案清單（路徑 + 各一句話說明）、驗證指令的實際輸出、
遇到的取捨決定（若有）。總長 ≤ 40 行。程式碼不用貼回，我會看檔案。
失敗時：回報卡在哪一步、錯誤訊息原文、你試過什麼。不要交「大概能動」的半成品。
```

## T3. 重構（agent: general-purpose ｜ model: sonnet ｜ effort: high；已定義好模式的批次套用可降 haiku）

適用：不改行為、改結構。與 T2 的差別：重構的驗收核心是「行為沒變」。

```
目標：重構 【範圍】：【改什麼結構，例如「把三處重複的權限檢查收斂成一個函式」】。
動機：【為什麼值得動它】。

不變式（最重要）：
- 對外行為完全不變：【怎麼證明——測試/實跑對照】
- 不順手修 bug、不順手改風格——看到問題記下來回報，不要動

模式定義：【改動前→改動後的具體範例一組；批次套用時這格必填】

驗收條件：
- 重構前先跑 【測試指令】 記下結果；重構後結果一致
- 改動檔案數與預告一致：【預計 N 個檔】

回報格式：改動清單（路徑 + 摘要）、前後測試輸出對照、
順手發現但沒動的問題清單（若有）。總長 ≤ 40 行。
失敗時：哪個檔改到一半發現模式不適用 → 停在那裡回報，不要硬套。
```

## T4. 研究（agent: general-purpose ｜ model: sonnet ｜ effort: medium；結論將供重大決策時升 opus）

適用：查資料、比較方案、事實查核。搭配 `skills/research/SKILL.md` 的輸出規範與 web search 策略。

```
目標：回答 【研究問題，一句話，可回答的形式】。
動機：【這個答案會拿來做什麼決策——這決定深度與立場】。

範圍與資料源：
- 先查內部：【memory/、outputs/ 的相關檔案路徑】
- 再查外部：web search 最多 【N，≤3】 輪（策略見 skills/research/SKILL.md）
- 時間範圍/地區/語言限制：【】

驗收條件：
- 結論可直接回答研究問題（不是「資料如下請自行判斷」）
- 每個事實主張附來源；區分 已知事實/合理推論/待驗證/高風險假設 四層
- 資料不足就明說不足，禁止硬湊

回報格式：完整筆記寫到 outputs/drafts/【檔名】.md（格式照 research skill 輸出規範）；
只回傳：結論 3 行以內 + 最關鍵的 3 個事實 + 檔案路徑。
```

## T5. 審查（agent: general-purpose ｜ model: sonnet ｜ effort: medium ｜ **必須 fresh context**）

適用：驗收別人（包括你自己派出去的工人）的產出。給它驗收材料，**不要給執行過程**，不要透露你希望的結果。

```
你是驗收者，之前沒有參與產出過程。獨立審查以下產出，不要預設它是對的。

受審產出：【檔案路徑】
背景一句話：【這是什麼任務的產出——只給必要背景，不給過程】

逐條驗收（每條標 pass / fail + 證據位置）：
- 【DoD 第 1 條，逐字抄自 Task Card】
- 【DoD 第 2 條】
- 【…】

另外檢查：
- 【產出類型對應的品質底線，抄自 JUDGMENT_RUBRICS R5 表格那一列】
- 內部矛盾、失效引用（提到的路徑/名稱是否真的存在——實際去驗證）

回報格式：逐條 pass/fail 表 + 每個 fail 的具體證據（檔案:行號 + 缺什麼）
+ 最後一行總判定：「可交付 / 需修正（列修正項）/ 需重做（說明根因）」。
不要客氣，fail 就是 fail；也不要超出清單自行加標準。
```

## 通用附註

- 派工時在 prompt 開頭補一句當前日期與專案根路徑，subagent 沒有你的環境感。
- 同時派多個互不依賴的工人：一次訊息裡並行派出，不要序列等待。
- 收到回報後的動作順序：先核對回報格式合約有沒有被遵守（沒遵守＝回報本身不可信）→ 再看結論 → 需要驗收的走 T5。
- 模板措辭可自行微調；**驗收條件與回報格式兩塊的結構不可刪**（改結構屬制度變更，見 `system/MAINTENANCE_PROTOCOL.md`）。
