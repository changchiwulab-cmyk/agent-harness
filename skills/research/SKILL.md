# Research Skill

## 用途
資料調查、產業分析、競品研究、技術評估、事實查核。

## 執行流程
1. 確認研究問題與範圍（從 Task Card 的 goal 取得）
2. 先查 project 內部資料（memory/、既有 outputs/）
3. 不足時再做 web search（最多 3 輪）
4. 整理成結構化筆記
5. 明確區分：已知事實 / 合理推論 / 待驗證 / 高風險假設
6. 輸出到指定位置

## 輸出格式
```markdown
# [研究主題]

## 結論
[一段話摘要]

## 已知事實
- ...

## 合理推論
- ...

## 待驗證
- ...

## 高風險假設
- ...

## 來源
- [來源1]
- [來源2]
```

## 工具/資源盤點的採用狀態定義

當研究任務涉及工具或資源盤點時，採用狀態使用以下四種值：

| 狀態 | 說明 |
|------|------|
| ✅ 已用 | 確認目前正在使用 |
| ✅ 推論已用 | 依脈絡合理推論在使用中，但未直接確認 |
| 🔍 待評估 | 尚未採用，值得評估是否導入 |
| ❌ 不適用 | 不符合需求或已排除 |

## 品質標準
- 每個事實主張都要有來源
- 推論要標明推論依據
- 不確定的資訊標記 [待驗證]
- 不把搜尋結果直接複製貼上，要消化重組

## Web search 策略

| 規則 | 說明 |
|------|------|
| 精準優先 | 第 1 輪查詢前先列出 2-3 組關鍵字，挑命中率最高的；不做廣域抓取 |
| 保留 1 輪備用 | `max_web_searches: 3` 的任務只用前 2 輪，第 3 輪保留給驗證或補洞 |
| 內部優先 | 先掃 `memory/`、`outputs/reports/`、既有 drafts/，內部命中後才出網 |
| Rate limit 對策 | 命中 rate limit 立即 checkpoint + 以已得結果完稿，不重試燒 token |

> SEC-03 對照：20260404-S01 任務 3/3 輪用滿後遭遇 rate limit（`logs/errors/2026-04-04_20260404-S01_error.md`）。
> 本策略的核心是把第 3 輪當保險，而非預設用滿。

## 常見失敗模式
- 搜尋結果不足就硬湊結論 → 應明確說「資料不足」
- 把單一來源當權威 → 應交叉比對
- 混淆事實與推論 → 應明確分類
- web search 預設用滿 3 輪 → 應留 1 輪備用（見上）
