# Research Skill

## 用途
資料調查、產業分析、競品研究、技術評估、事實查核。

## 執行流程
1. 確認研究問題與範圍（從 Task Card 的 goal 取得）
2. 先查 project 內部資料（memory/、既有 outputs/）
3. 不足時再做 web search（最多 3 輪）
4. 整理成結構化筆記
5. 明確區分：已知事實 / 合理推論 / 待驗證 / 高風險假設
6. 輸出到指定位置

## 知識截止與 web search 優先順序
- **Opus 4.7 知識截止 = 2026-01**（見 system/MODEL_POLICY.yaml / D005）
- 若研究主題涉及 **2026-02 之後的事實**（新發佈的產品、價格、法規、市場數據、模型版本），
  **預設優先使用 WebSearch**，不可僅憑模型內部知識作答
- 若 Task Card 的 `max_web_searches: 0` 但主題落在 cutoff 之後，
  在輸出的「高風險假設」區塊明確標註「資料可能過時，建議補 web search」
- 歷史性或原理性主題（不隨時間變化），可優先使用內部知識，web search 作為交叉驗證

## 輸出格式
```markdown
# [研究主題]

## 結論
[一段話摘要]

## 已知事實
- ...

## 合理推論
- ...

## 待驗證
- ...

## 高風險假設
- ...

## 來源
- [來源1]
- [來源2]
```

## 工具/資源盤點的採用狀態定義

當研究任務涉及工具或資源盤點時，採用狀態使用以下四種值：

| 狀態 | 說明 |
|------|------|
| ✅ 已用 | 確認目前正在使用 |
| ✅ 推論已用 | 依脈絡合理推論在使用中，但未直接確認 |
| 🔍 待評估 | 尚未採用，值得評估是否導入 |
| ❌ 不適用 | 不符合需求或已排除 |

## 品質標準
- 每個事實主張都要有來源
- 推論要標明推論依據
- 不確定的資訊標記 [待驗證]
- 不把搜尋結果直接複製貼上，要消化重組

## 常見失敗模式
- 搜尋結果不足就硬湊結論 → 應明確說「資料不足」
- 把單一來源當權威 → 應交叉比對
- 混淆事實與推論 → 應明確分類
