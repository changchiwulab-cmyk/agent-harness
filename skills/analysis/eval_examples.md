# Analysis Skill — 評估範例

好的 analysis 輸出 vs 壞的 analysis 輸出的對比，用於維持品質標準。

---

## 範例一：框架優化方向決策（來自 20260414-A01）

### 情境
評估 Agent Harness v2 的完整度與優化程度，找出關鍵問題並排定優先修復順序。

---

### ✅ 好的 Analysis 輸出

```markdown
## 結論與建議
最優先修復：Approval Pipeline 從未實際執行（outputs/reports/ 空白）。
其次：skill_type 白名單多處維護導致 regression，建立 SKILL_TYPES.yaml 解決根因。
第三：QUICKSTART.md 缺失，使用者體驗進入門檻過高。

## 選項比較

### 選項 A：先修框架文件（QUICKSTART.md、CLAUDE.md 瘦身）
| 維度       | 評估             | 依據                            |
|-----------|-----------------|-------------------------------|
| 價值       | 中               | 降低新任務啟動摩擦               |
| 成本       | 低（~1K token）  | 純文件修改                      |
| 風險       | 低               | 不影響現有執行流程               |
| 可行性     | 高               | 無外部依賴                      |
| 執行難度   | 低               | 一人即可完成                    |
| 預期回報   | 中               | 減少每次任務的入門時間            |
| 一人公司適配度 | 高            | 自己用自己建，成本極低           |

### 選項 B：先修 Approval Pipeline（讓輸出真正升格）
| 維度       | 評估              | 依據                                      |
|-----------|------------------|------------------------------------------|
| 價值       | 高                | 框架的核心承諾（Real Artists Ship）         |
| 成本       | 低（~2K token）   | 建立 approval record + 複製檔案             |
| 風險       | 低                | 不修改核心邏輯，只是執行已設計的流程         |
| 可行性     | 高                | 機制已存在，只需實際操作                    |
| 執行難度   | 低                | 步驟清晰，無技術難度                        |
| 預期回報   | 高                | 一次實際跑通 > 十次文件優化                 |
| 一人公司適配度 | 高             | 驗證框架可靠性的最直接方式                  |

### 選項 C：不做（維持現狀）
| 維度       | 評估 | 依據                          |
|-----------|------|------------------------------|
| 價值       | 零   | 問題持續積累                  |
| 成本       | 零   | 無需投入                      |
| 風險       | 中   | regression bug 持續影響 CI    |
| 可行性     | 高   | 技術上可行                    |
| 執行難度   | 零   | 無需執行                      |
| 預期回報   | 負   | 技術債增加                    |
| 一人公司適配度 | 低 | 框架問題會拖慢實際業務執行     |

## 高風險假設
- **假設**：Approval Pipeline 只是流程問題，不是設計問題
  - 如果不成立：可能需要重新設計 draft_first 流程的 UX

## 待驗證
- Approval Pipeline 跑通後，是否真的降低了後續任務的信心門檻

## 建議下一步
1. **立即**：執行一次完整 Approval 流程（B 選項，最高 ROI）
2. **本週**：修復 regression bug（SKILL_TYPES.yaml）
3. **下週**：QUICKSTART.md + CLAUDE.md 瘦身（A 選項）
```

**為什麼這是好的 analysis：**
- 每個選項都有七維評估，且有具體依據（不是模糊的「高/中/低」）
- 包含「不做」選項
- 結論明確排序，可追溯到分析內容
- 識別了高風險假設
- 建議下一步具體、可執行

---

### ❌ 壞的 Analysis 輸出

```markdown
## 分析

框架有幾個問題：
- 文件太多了
- Approval Pipeline 沒有用
- 有 bug

## 建議
建議先修文件，然後再修 bug，最後處理 Approval Pipeline。
```

**為什麼這是壞的 analysis：**
- 沒有結構化的選項比較（每個問題直接跳到結論）
- 「文件太多」沒有量化（多多少？影響是什麼？）
- 沒有「不做」選項
- 建議順序的理由不清楚（為什麼文件優先於 bug？）
- 沒有高風險假設和待驗證項目

## 範例二：一人公司商業模式決策

### 情境
決定一人公司在台灣市場應優先發展哪種服務類型（顧問 vs SaaS vs 培訓）。

---

### ✅ 好的 Analysis 輸出關鍵特徵

1. **量化成本**：「顧問模式：前 3 個月投入 ~40 小時/月建立案例，每案預期 NT$80K–120K」
2. **識別一人公司限制**：「SaaS 需要持續技術維護，若無法接受 1 週內無法回應用戶，此模式不適合單人」
3. **明確「不做」選項**：「維持現有接案模式，收入上限約 NT$200K/月，成長天花板明確」
4. **高風險假設標記**：「假設台灣 AI 顧問市場願意接受 NT$15K/天的費率 — 若不成立，整體獲利模型需重算」
5. **結論可追溯**：「建議優先顧問模式，原因：(1) 啟動成本最低，(2) 台灣市場驗證案例存在（20260404-S01），(3) 可在 6 個月內達到損益兩平」

---

## 快速判斷：這份 analysis 及格嗎？

| 檢查項 | 通過條件 |
|--------|---------|
| 選項完整性 | 包含「不做」選項 |
| 評估結構 | 每個選項都有六維評估 |
| 量化程度 | 成本/風險有具體數字或區間 |
| 結論可追溯 | 結論中引用分析中的具體依據 |
| 一人公司視角 | 每個選項評估了執行可行性 |
| 高風險假設 | 列出至少一個關鍵假設及其不成立的影響 |

全部通過 → 輸出合格。任一不通過 → 補充後再輸出。
