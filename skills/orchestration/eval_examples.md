# Orchestration Skill — 評測範例

## 好的輸出範例

> 題目：研究越南 AI 工具市場後，撰寫一份進入策略提案（research → writing）

```markdown
# 越南 AI 工具市場進入策略（編排彙整）

## 結論
建議以「客服自動化」為切入點先行試點：research 子任務顯示該場景採用率最高、
writing 子任務據此產出的提案在成本與落地難度上最可行。

## 子任務結果
- s1（research）：越南主流 AI 工具採用現況與市佔；output：outputs/drafts/vietnam-ai-tools-research.md
- s2（writing）：依 s1 撰寫的進入策略提案；output：outputs/drafts/vietnam-entry-proposal.md

## 彙整與交叉檢查
s2 的每項策略主張都對應到 s1 的具體數據；未發現提案引用 s1 未涵蓋的假設。

## 待驗證
- 本地法規對 AI 客服的資料落地要求：需法務確認

## 來源對應
- 「客服自動化採用率最高」← s1 §已知事實
- 「試點成本估算」← s2 §成本評估
```

**為什麼好**：結論先行；每個子任務有獨立 output 路徑；彙整做了交叉檢查；結論可追溯到子任務。

---

## 壞的輸出範例

> 同一題目的錯誤輸出

```markdown
越南市場很有潛力，AI 工具越來越普及。我們研究了一下，然後寫了一份提案，
建議進入這個市場。團隊覺得客服方向不錯，應該會成功。各種工具都可以考慮，
之後再看情況調整。
```

**哪裡錯**：
- 沒有結論先行，整段是模糊敘述
- 沒有區分子任務、沒有 output 路徑、無法追溯
- 沒有交叉檢查，也沒有標記待驗證
- 把編排寫成一段散文，等於繞過了子任務治理

---

## 判斷標準

| 項目 | 通過 | 不通過 |
|------|------|--------|
| 結論先行 | 第一段就是整體結論 | 先鋪陳再含糊帶過 |
| 子任務可追溯 | 每個子任務有 output 路徑 | 看不出子任務與來源 |
| 交叉檢查 | 彙整有檢查子任務一致性 | 直接拼貼或省略 |
| 治理 | 子任務各自有卡與 gate | 一段散文取代編排 |
