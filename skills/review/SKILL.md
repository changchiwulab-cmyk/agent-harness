# Review Skill

## 用途
品質審查、文件校對、邏輯檢查、風險評估、決策驗證、輸出回測。

## 執行流程
1. 確認審查對象與標準（從 Task Card 取得）
2. 載入被審查的內容
3. 依序執行檢查項：
   a. 格式與結構完整性
   b. 事實正確性（有來源的查來源，無來源的標記）
   c. 邏輯一致性（結論是否有依據支撐）
   d. 風險識別（遺漏、矛盾、過度承諾）
4. 產出審查報告
5. 明確列出：通過項 / 問題項 / 建議修改項

## 審查原則
- 不改原文，只提審查意見
- 問題要具體到段落或句子層級
- 區分「必須修改」與「建議修改」
- 不要只挑毛病，也要指出做得好的地方
- 對照 definition_of_done 逐條確認

## 輸出格式
```markdown
# 審查報告：[被審查文件名]

## 總體評估
[通過 / 有條件通過 / 需修改]

## 通過項
- ...

## 必須修改
- [位置] [問題描述] [建議]

## 建議修改
- [位置] [問題描述] [建議]

## Definition of Done 逐條確認
- [ ] 條件1：[通過/未通過] [說明]
- [ ] 條件2：[通過/未通過] [說明]
```

## 品質標準
- 每個問題都有具體位置與修改建議
- 不遺漏 definition_of_done 中的任何條件
- 嚴重問題不能被降級為建議

## 常見失敗模式
- 審查太寬鬆（全部說 OK）→ 至少找出 1 個可改善的地方
- 審查太嚴苛（雞蛋裡挑骨頭）→ 專注在影響結論的問題
- 只看格式不看內容 → 三層都要跑
- 自己的輸出審查自己（循環驗證）→ review 應針對不同 session 的產出
